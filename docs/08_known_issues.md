# 08. 既知の課題・注意点・改善候補

このファイルは、現状コードを読んだ時点での注意点・将来の改善候補をまとめたものです。
不具合修正や仕様変更の依頼時に、優先度付けの参考にしてください。

## 1. 文字化けコメント

`src/controller_main.cpp` と `src/crawler_main.cpp` の日本語コメントが、環境によって文字化けして表示されています。

### 影響

- 仕様理解や保守性が低下する。
- 今後の差分レビューがしづらい。

### 改善案

- ソースファイルを UTF-8 に統一する。
- 文字化けコメントを本ドキュメントに基づいて書き直す。

## 2. `ControlData` が固定長型ではない

現在:

```cpp
struct ControlData {
  int x;
  int y;
  int maxSpd;
};
```

### 影響

- ESP32 同士では通常問題ないが、`int` サイズ・アライメントが暗黙前提。
- 将来のプロトコル拡張で互換性管理が難しい。

### 改善案

- `int16_t` など固定長型へ変更。
- `version`, `checksum`, `sequence` を追加。
- 送受信値の範囲チェックを追加。

## 3. ESP-NOW がブロードキャスト・送信元未検証

現状はブロードキャスト送信で、受信側は送信元 MAC アドレスを確認していません。

### 影響

- 同一チャンネル・同一ペイロード形式の送信機が複数あると混信する可能性がある。
- 意図しない操作データを受け付ける可能性がある。

### 改善案

- クローラ側 MAC アドレスを指定してユニキャスト送信する。
- 受信側で送信元 MAC アドレスを許可リスト検証する。
- ペアリング ID をペイロードに含める。

## 4. コントローラ側で MiniJoyC 未検出時の送信データが明示初期化されていない

`data` はグローバル変数のため C++ の規則上ゼロ初期化されますが、コード上は明示されていません。
MiniJoyC 未検出時は `x`, `y` を更新しないため、初期値依存になります。

### 改善案

- `ControlData data = {0, 0, 2000};` のように明示初期化する。
- MiniJoyC 未検出時は毎ループ `data.x = 0; data.y = 0;` を設定する。

## 5. `maxSpd` の受信値検証がない

クローラ側は受信した `incomingData.maxSpd` をそのまま `maxSpeed` に代入します。

### 影響

- 異常値を受信した場合、想定外のサーボ速度が出る可能性がある。

### 改善案

```cpp
maxSpeed = constrain(incomingData.maxSpd, 0, 6000);
```

または仕様として `1000..6000` に制限する。

## 6. サーボ初期化結果を確認していない

`unLockEprom()`, `WheelMode()`, `EnableTorque()` の戻り値を確認していません。

### 影響

- サーボ未接続・ID 不一致・通信不良を検出しにくい。

### 改善案

- 初期化 API の戻り値を確認し、画面またはシリアルに表示する。
- 失敗した ID を記録して Avatar 表情や画面で通知する。

## 7. 起動ごとに EPROM ロック解除・ホイールモード設定を行う

サーボ設定が不揮発領域へ書き込まれる場合、起動ごとの設定変更が適切か確認が必要です。

### 改善案

- 初回セットアップ用コマンドと通常起動を分離する。
- 現在モードを読み取れる場合は、必要な場合のみ設定する。
- 設定後に `LockEprom()` を行うか検討する。

## 8. MiniJoyC I2C 読み取りのエラー処理が弱い

`M5HatMiniJoyC::readBytes()` は `requestFrom()` 後に実際の受信バイト数を確認せず、`read()` しています。

### 影響

- I2C 通信エラー時に不正値を読み取る可能性がある。

### 改善案

- `requestFrom()` の戻り値を確認する。
- 読み取り失敗時は直前値またはニュートラル値を返す。
- エラー回数を画面表示する。

## 9. M5HatMiniJoyC 内で `_wire` と `Wire` が混在

`getI2CAddress()` と `getFirmwareVersion()` では `_wire->requestFrom()` ではなく `Wire.read()` を使用しています。

### 影響

- 現状は `Wire` を使うため問題化しにくいが、別 I2C バスを使う場合に不具合となる。

### 改善案

- `Wire.read()` を `_wire->read()` に統一する。

## 10. コントローラのボード定義

`platformio.ini` では環境名が `m5stick-c-plus2` ですが、ボード定義は `board = m5stick-c` です。

### 影響

- M5StickC Plus2 固有の設定が必要な場合に不一致が問題になる可能性があります。

### 改善案

- PlatformIO が M5StickC Plus2 用ボード定義を提供している場合は変更を検討する。
- 現状動作している場合は、変更前に表示・ボタン・USB 書き込みの確認を行う。

## 11. 右側・上側サーボの符号反転がハードウェア依存

クローラ側で以下の符号反転があります。

```cpp
speedR = -speedR;
sts.WriteSpe(SERVO_TOP_LEFT_ID, -speedL, 0);
sts.WriteSpe(SERVO_TOP_RIGHT_ID, -speedR, 0);
```

### 影響

- サーボ取り付け方向や配線を変更すると、前進/後退/旋回方向が変わる。

### 改善案

- サーボごとの方向係数を定義する。

```cpp
const int DIR_LEFT = 1;
const int DIR_RIGHT = -1;
const int DIR_TOP_LEFT = -1;
const int DIR_TOP_RIGHT = 1;
```

- キャリブレーション手順をドキュメント化する。

## 12. 画面更新で毎回全消去している

コントローラ側は送信ごとに `fillScreen(BLACK)` して画面を書き直しています。

### 影響

- ちらつきや描画負荷が出る可能性がある。

### 改善案

- 変更がある行だけ再描画する。
- スプライト描画を使う。
- 表示更新周期を送信周期とは別にする。

## 13. テストコードがない

`test/README` はありますが、現状のユニットテストや結合テストはありません。

### 改善案

- `ControlData` のサイズ検証テスト。
- ジョイスティック値から `x/y` への変換テスト。
- `joyX/joyY` から `speedL/speedR` への変換テスト。
- `maxSpeed` 切替ロジックのテスト。
